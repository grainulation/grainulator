grainulator.app Demo Redesign

Decision brief from 8 research sprints, 4 witness passes, 1 blind-spot, 1 calibration — 150+ claims

Executive Summary

The current grainulator.app demo is broken: the try button appears before the model loads (CSS display:inline-flex overrides HTML hidden), claim cards are invisible (.drifting animation replaces card-in), and pressing Enter rebuilds identical DOM. These are now fixed in production.

The deeper problem: downloading a 200-900MB model for a single-use demo is bad UX. Users wait 30-120 seconds, get one response from a 360M-parameter model, and can’t continue a conversation. The real product (wheat in Claude Code) runs on Opus and is vastly more capable.

The recommendation: Ship in 3 phases. Phase 0 (1 day) ships a video of the real product + fixes remaining bugs. Phase 1 (6.5 days) builds the cascade + conversational demo + handoff. Phase 2 (2 days) adds engagement polish. Critical gate on Day 2: if SmolLM2-360M + XGrammar produces embarrassing output, pivot to library-only conversational (fuzzy-matched pre-generated demos for all turns).

Phase 0: Video + Bugs Tested: 1 day

Ship before any cascade work. Highest ROI per hour.

Rx1 — Record a video of the real product

A 30-second screen recording of a real wheat sprint running in Claude Code. Shows the ACTUAL product: multi-agent research, real evidence tiers, compiler catching conflicts, decision brief output. Embed above the fold on the landing page. The video creates desire; the demo creates familiarity.

Why: Interactive demos convert 1.8x better than passive content (x019), but only when the demo is good. A 360M model demo risks showing users a crippled version of the product. The video shows the real thing.

Format: MP4, autoplay muted, loop. Dark terminal aesthetic matches the lava lamp. No narration needed — the terminal output speaks for itself.

Rx2 — Zero-dep analytics

No third-party analytics service. Measure via proxy signals + local engagement:

  • Aggregate: GitHub Pages traffic (Settings → Traffic, 14-day window). Archive weekly via GitHub Actions cron job.
  • Install conversion: npm weekly download count via npm-stat.com — the real conversion metric. Public, no code needed.
  • Repo engagement: Stars, forks, watchers via GitHub API.
  • Per-user engagement: IndexedDB in service worker tracks session events (demo_enter, question_submitted, compile_run, handoff_click). Powers gamification badges locally. Data stays on device — zero privacy concern.

Why: Aggregate analytics is unknowable on static hosting without a third party. Accept this. The true conversion metric is npm installs/week, which is public.

Rx3 — Fix mobile

Currently: install button hidden on mobile (line 214: .cta-install{display:none}). 40-60% of traffic sees no conversion path (x008, x009).

Fix: Show install button on mobile. Change it to copy /plugin install grainulator to clipboard with a toast “copied — paste in Claude Code.” The “explore demo” button loads Track A (works without WebGPU).

Rx4 — Feature flag

Add ?cascade=off URL parameter. When present, skip all cascade logic and use the current simple demo (Track A + local model if available). This is the rollback path if anything breaks post-deploy (x014).

Phase 1: MVP Demo Built: 6.5 days

The core redesign. Gate on Day 2: test XGrammar + 360M output quality.

Rx5 — Pre-generated demo library (50 questions)

Build-time artifact: 50 pre-computed question-answer sets generated by Claude. Each is a complete sprint: question, 4-6 typed claims with varied evidence tiers, compiler output with tensions/warnings. ~20KB gzipped. Zero runtime dependencies.

Build step: Run a Claude script that generates 50 diverse engineering research questions and full claim sets. Store as JSON inlined in the HTML or loaded from a demos.json file cached by the service worker. Topics cover: architecture decisions, technology comparisons, scaling strategies, migration planning, build-vs-buy, team processes.

Fuzzy matcher: ~150 lines of custom Levenshtein + word overlap scoring. When user types a question and hits Enter (before model is ready), match to closest library entry in <1ms. Display the matched demo instantly.

Template fallback: For questions matching nothing: extract keywords, fill placeholder claims labeled honestly as “Placeholder claims — local model loading for live analysis...”

Why zero-dep? No CORS, no privacy disclosure, no free tier limits, no outage risk. Claude-quality demos at build time > small-model inference at runtime. Everything runs locally.

Rx6 — 3-layer zero-dep cascade

All layers run locally. No external calls at inference time:

Layer 0 (INSTANT, 0ms):  Fuzzy-matched pre-generated demo from library
Layer 1 (CACHED, ~2s):   WebLLM model from Cache Storage (return visitors)
Layer 2 (DOWNLOADING):   WebLLM model via CDN in background (first visit)

On Enter: If local model is ready (Layer 1), run live inference. Otherwise, fuzzy-match to the pre-generated library (Layer 0). The user gets an instant, Claude-quality response while the model downloads silently.

Return visitor fast path: Check Cache Storage on page load. If model is cached, skip pre-generated content — enable input immediately. Return visits feel like opening a local app (~2s).

Context continuity: When the model finishes downloading mid-session, inject accumulated claims into the system prompt. The local model adds to (never replaces) library-generated claims.

Debug panel: ?debug=cascade shows which layer was selected, match score, and allows force-selecting layers.

Rx7 — 2-3 EXPLORE turns with XGrammar

Critical gate: On Day 2, test SmolLM2-360M with XGrammar constrained decoding. If output is embarrassing, pivot to library-only conversational (fuzzy-matched pre-generated demos for all interactions).

Turn model: EXPLORE only. No CHALLENGE turns (360M cannot self-critique, MT-Bench 3.66/10 — w2103). Each turn adds 3-5 claims to the accumulated set. Claim IDs generated in application code, not model output (w2110: GSM8K 7.4%).

Context window: System prompt (~500 tokens) + compressed claim summary (~50 tokens/claim, max 15 claims = 750 tokens) + last 2 user turns. Total: ~2000 tokens of context, leaving ~6000 for generation.

XGrammar enforcement: Use WebLLM’s JSON mode with schema constraint. Do NOT fall back to regex parsing. If generation fails, show error toast, don’t silently load Track A (x022).

State: Persist in localStorage: claims[], conversationHistory[], compilationResult. 7-day expiry. Resume/fresh prompt on return.

Rx8 — Follow-up chips after compile

After compilation, the verdict surfaces gaps. Convert gaps into clickable action chips:

  • Missing risk claims → chip: “Find risks”
  • Type monoculture → chip: “Diversify evidence”
  • Low evidence tier → chip: “Strengthen evidence”
  • Tension detected → chip: “Resolve tension between r001 and r003”

Clicking a chip submits it as the next EXPLORE turn. This drives the conversation forward without requiring the user to know what to ask.

Rx9 — Dual handoff

After compile or at turn 3+, show two handoff buttons:

  1. “Open in Claude Code” — Constructs claude-cli://open?q=<prompt> with a human-readable summary (~1300 chars). The prompt includes the question, a 2-3 sentence summary of findings, and an instruction to continue with wheat. Fits within Windows 2K limit (w2202).
  2. “Copy command” — Copies a longer claude "..." command to clipboard via Clipboard API. Universal fallback, no install requirement.

Not installed? If claude-cli:// fails (no Claude Code installed), show a toast: “Install Claude Code first” with link, then the clipboard fallback.

Payload format:

I explored a research question in the grainulator.app demo
and want to continue with the full wheat plugin.

Question: [user's question]
Key findings: [2-3 sentence summary from compilation verdict]
Claims so far: [count] claims across [topics] topics
Confidence: [score]/100

Please install the wheat plugin (/plugin install grainulator)
and continue this research sprint from where I left off.

Rx10 — Handoff nudges

Turn 3: Subtle link below input: “Want deeper research? Continue in Claude Code →”

Turn 5: Prominent banner: “You’ve explored 5 questions. For multi-source evidence, deeper analysis, and production-grade sprints, open in Claude Code.” With both handoff buttons.

Never block: User can always keep going. The nudge is a suggestion, not a wall.

Phase 2: Engagement Polish Built: 2 days

Only after Phase 1 is validated via analytics.

Rx11 — Border glow peek

Replace fully opaque demo overlay with one that has 8px margin and border-radius:12px, letting the lava lamp shader bleed through the edges. The shader IS the brand — hiding it completely in the demo loses the visual signature (w2403).

During model download: Use CSS-only fallback gradient (already in codebase, lines 266-279) for the border peek. Resume real shader only after model is cached. Avoids GPU contention (x020).

Rx12 — Shader blob progress

The shader already has u_preloadProgress uniform driving blob growth (line 512-516). Currently growFactor is 0.3 — amplify to 0.6 for visible expansion through the border glow peek. 15-minute change (w2406).

Rx13 — 3 demo topics

Replace single PostgreSQL vs CockroachDB demo with 3 topics that showcase different compiler behaviors:

  1. Tension demo: “Monorepo vs polyrepo for a 50-engineer org” — r001 (recommendation: monorepo) conflicts with r003 (risk: CI bottleneck). Compiler flags tension.
  2. Happy path: “When to adopt AI coding assistants” — balanced claim types, good evidence, clean compile at 75/100.
  3. Type monoculture: “Build vs buy for authentication” — all recommendations, no risks. Compiler flags monoculture warning at 55/100.

Selection: Topics that are universally relevant to the target audience (engineers), inherently controversial (tension detection), and time-sensitive (motivation for real research) — x013.

Rx14 — Attract mode auto-play

While model downloads, the demo performs itself: question types in (3s) → claims animate one by one (3s) → compiler runs 7 passes (3s) → verdict appears (2s) = ~11s per cycle. Plays 1 full rotation across all 3 topics (~33s), then stops with a “Replay” button (w2408).

Interrupt: Clicking the input or any claim card during auto-play pauses it immediately. If model is ready, input enables.

Reduced motion: Skip typewriter and card animations. Show claims as static list. Respect prefers-reduced-motion (w2412).
